# Hook0 Documentation

## Diagrams — Mermaid Design System (dark/light mode)

- NEVER use ASCII art in documentation. Always use Mermaid diagrams.
- Docusaurus has `@docusaurus/theme-mermaid` configured with `{ light: 'base', dark: 'dark' }`.
- NEVER use `style NodeX fill:#hex` in mermaid blocks. Use `classDef` + `:::class` syntax instead.
- `classDef` lines in .md files set **light mode colors inline** (fill, stroke, color). This makes light mode work out of the box.
- **Dark mode** is handled by a JS client module (`documentation/src/mermaid/theme-switcher.js`) that mutates inline SVG styles on theme change. CSS `!important` cannot reliably override Mermaid's inline `style` attributes, so JS is required.
- CSS variables for mermaid colors are kept in `documentation/src/css/custom.css` for reference but are NOT used for node styling.
- Prefer `flowchart LR` for data flows, `stateDiagram-v2` for state machines.

### Semantic node classes

| Class | Semantic role | Light mode | Dark mode |
|-------|--------------|------------|-----------|
| `:::external` | External systems, inputs, API clients | Blue (#dbeafe bg) | Deep blue (#1e3a5f bg) |
| `:::hook0` | Hook0 components, infrastructure | Green (#dcfce7 bg) | Deep green (#052e16 bg) |
| `:::customer` | User/customer endpoints, outputs | Orange (#ffedd5 bg) | Deep orange (#431407 bg) |
| `:::processing` | Internal processing, middleware | Purple (#ede9fe bg) | Deep purple (#2e1065 bg) |
| `:::danger` | Error/failure states | Red (#fee2e2 bg) | Deep red (#450a0a bg) |

### Template for new mermaid blocks

```
flowchart LR
    A["Input"]:::external --> B["Hook0"]:::hook0
    B --> C["Output"]:::customer
    B -.- D["Internal"]:::processing

    classDef external fill:#dbeafe,stroke:#60a5fa,color:#1e3a5f
    classDef hook0 fill:#dcfce7,stroke:#4ade80,color:#14532d
    classDef customer fill:#ffedd5,stroke:#fb923c,color:#7c2d12
    classDef processing fill:#ede9fe,stroke:#a78bfa,color:#3b0764
    classDef danger fill:#fee2e2,stroke:#f87171,color:#7f1d1d
```

### Rules
1. ALWAYS add `classDef` lines at the bottom of the mermaid block (before `click` lines).
2. ALWAYS apply `:::class` to every node definition.
3. The `classDef` values are the light-mode colors. The JS client module (`src/mermaid/theme-switcher.js`) handles dark mode by mutating inline styles.
4. For subgraph-level styling, use `class SubgraphName hook0` syntax.
5. Only include the `classDef` lines for classes actually used in the diagram.

## Internal linking rules

When writing or editing documentation pages, link Hook0 concepts to their concept pages on first meaningful mention per page.

### Concept-to-page mapping

| Concept mention | Link target |
|----------------|-------------|
| organization / organizations | `/concepts/organizations` |
| application / applications | `/concepts/applications` |
| event / events (Hook0 events, not general) | `/concepts/events` |
| event type / event types | `/concepts/event-types` |
| subscription / subscriptions | `/concepts/subscriptions` |
| label / labels (Hook0 labels) | `/concepts/labels` |
| request attempt / request attempts | `/concepts/request-attempts` |
| application secret / application secrets | `/concepts/application-secrets` |
| service token / service tokens | `/concepts/service-tokens` |
| metadata (Hook0 metadata concept) | `/concepts/metadata` |

### Linking rules

1. Link the **first meaningful mention** of each concept per page (1-3 concept links per page is typical, more for long pages).
2. **Do not link** in code blocks, Mermaid diagrams, or table cells that are already links.
3. **Do not self-reference**: never link "subscriptions" on the subscriptions concept page itself.
4. **Do not over-link**: if a concept appears 10 times on a page, only the first prose mention should be a link.
5. Every **new page** must have at least 2 internal links to other documentation pages.
6. Every **new page** should be linked FROM at least 1 existing page (e.g., added to a "Next steps" or "Further reading" section).

## SEO rules

### Frontmatter requirements

Every page MUST have frontmatter with:
- `title`: specific, includes the primary keyword naturally
- `description`: 150-160 characters, action-oriented ("How to...", "Learn...", "Set up...")
- `keywords`: array of 3-7 relevant keywords

### Headings

- Use **sentence case** for all headings (e.g., "Getting started with Hook0", not "Getting Started With Hook0")
- The `title` frontmatter field is used as the page `<title>` and should be distinct from the `# H1` heading

### Page structure

- Include a **"Next steps"** or **"Further reading"** section at the bottom of every page with 2-4 internal links
- New comparison pages must target a specific search query cluster (e.g., "Svix vs Hook0" targets users searching for Svix alternatives)

### Technical SEO

- The documentation sitemap is auto-generated by Docusaurus -- do not manually create one
- `trailingSlash: false` is configured in `docusaurus.config.js` -- Netlify `_redirects` handle the 301s
- Pages that should not be indexed: use `robots: 'noindex'` in frontmatter

### Content quality signals

- Descriptions should be action-oriented: "How to verify webhook signatures with HMAC" is better than "Webhook authentication documentation"
- Avoid thin pages: every page should provide substantial, unique content
- Use schema.org structured data (`application/ld+json`) on comparison pages (already done via `<Head>` component)
