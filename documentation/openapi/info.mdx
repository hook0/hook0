# Hook0 API Reference

Welcome to the Hook0 API reference documentation. This comprehensive guide provides detailed information about all available API endpoints, request/response formats, and authentication methods.

## Base URL

```
https://app.hook0.com/api/v1
```

For self-hosted instances, replace with your domain:

```
https://your-domain.com/api/v1
# Or http://localhost:8001/api/v1 for testing locally
```

## Authentication

Hook0 uses **API tokens** for authentication. All API requests must include an `Authorization` header:

```http
Authorization: Bearer {YOUR_TOKEN}
```

Manage API tokens for programmatic access.

## Rate Limiting

API requests are rate-limited to ensure fair usage:

- **Global limit**: 1,000 requests/second
- **Per-IP limit**: 100 requests/second
- **Per-token limit**: 10 requests/second

Rate limit headers are included in responses:

- `X-RateLimit-Limit`: Maximum requests allowed
- `X-RateLimit-Remaining`: Requests remaining
- `X-RateLimit-Reset`: Time when limit resets

## Error Handling

Hook0 API uses standard HTTP status codes:

- `200 OK`: Successful request
- `201 Created`: Resource created successfully
- `400 Bad Request`: Invalid request parameters
- `401 Unauthorized`: Missing or invalid authentication
- `403 Forbidden`: Insufficient permissions
- `404 Not Found`: Resource not found
- `429 Too Many Requests`: Rate limit exceeded
- `500 Internal Server Error`: Server error

Error responses include a JSON body:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid request parameters",
    "details": {
      "field": "email",
      "reason": "Invalid email format"
    }
  }
}
```

## Pagination

List endpoints use **cursor-based pagination**. Two query parameters control the page:

- `pagination_cursor` (optional): an opaque, base64-encoded token returned by Hook0 in the previous response's `Link` header. Clients **must not** parse or modify this value — treat it as a black box.
- `limit` (optional): number of items per page. **Default: 100. Min: 1. Max: 100.** Out-of-range values return HTTP 400.

Pagination metadata is returned via the `Link` HTTP header ([RFC 8288](https://www.rfc-editor.org/rfc/rfc8288)):

- `Link: <…>; rel="next"` — URL of the next page. Absent on the last page.
- `Link: <…>; rel="prev"` — URL of the previous page. Absent on the first page.

The next-link URL preserves the client's `limit`, so you do not need to re-append it when following the cursor.

To walk the full result set, follow `rel="next"` until no `next` link is present.

### Example

Request:

```http
GET /api/v1/event_types?application_id=11111111-1111-1111-1111-111111111111&limit=50
Authorization: Bearer {YOUR_TOKEN}
```

Response (truncated):

```http
HTTP/1.1 200 OK
Content-Type: application/json
Link: <https://app.hook0.com/api/v1/event_types?application_id=11111111-1111-1111-1111-111111111111&pagination_cursor=eyJkYXRlIjoiMjAyNi0wNC0yOFQxMjozNDo1Ni43ODlaIiwibmFtZSI6ImlhbS51c2VyLmNyZWF0ZWQiLCJkaXJlY3Rpb24iOiJmb3J3YXJkIn0%3D&limit=50>; rel="next"

[
  { "event_type_name": "iam.user.created", ... },
  { "event_type_name": "iam.user.updated", ... }
]
```

Follow the next page:

```http
GET /api/v1/event_types?application_id=11111111-1111-1111-1111-111111111111&pagination_cursor=eyJkYXRlIjoi…&limit=50
Authorization: Bearer {YOUR_TOKEN}
```

When the response no longer includes a `Link: rel="next"` header, you have reached the last page.

### Cursor opacity and security

The cursor is **a position hint, not an authorization token**. It encodes only the row to start from (a timestamp plus a tiebreaker). Authentication and per-tenant scoping are enforced independently by your [API token](#authentication) and the always-present `application_id` filter — a tampered cursor cannot leak data across tenants, only narrow the result.

Do not store cursors long-term. They are stable as long as the underlying row exists, but they are not contractual identifiers.

### Endpoint coverage

Cursor pagination is enabled on:

- `GET /api/v1/event_types`
- `GET /api/v1/subscriptions`
- `GET /api/v1/request_attempts`

Default page sizes currently differ between endpoints (100 for `event_types` and `subscriptions`, 50 for `request_attempts`). They will be unified to 100 in a future release. Until then, always rely on the `Link` header rather than assuming a fixed page size.

## Webhooks

When subscribing to events, Hook0 will send webhooks with:

- **Signature verification**: HMAC-SHA256 signature in `X-Hook0-Signature` header
- **Retry logic**: Automatic retries with a fixed retry schedule
- **Idempotency**: Event IDs ensure exactly-once delivery
