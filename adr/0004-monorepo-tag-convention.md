# 4. Monorepo tag convention

Date: 2026-05-09

## Status

Accepted

## Context

Hook0 is a monorepo that ships several independently consumable packages: `api`, `output-worker`, `cli`, `play`, `frontend`, `mcp` (MCP server), and the `sdk` (Rust + TypeScript clients shipped together).

Until now, releases used a single coupled flow: a `release:patch` job bumped `api`, `output-worker` and `frontend` together under one tag `vX.Y.Z`. The CLI and Play services lived at `0.1.0` indefinitely and the MCP crate had a one-off manual tag (`hook0-mcp-v1.0.0`). The SDK had its own dedicated flow with the prefix `sdk-vX.Y.Z`.

This forced unrelated bumps (a CLI hotfix had to wait for an API release window) and produced confusing user-facing version mismatches between components.

## Decision

We adopt a per-package tag convention using the slash separator:

```
<package>/v<major>.<minor>.<patch>[-<pre>]
```

Examples: `api/v1.0.1`, `cli/v1.0.1`, `frontend/v2.3.0`, `mcp/v1.0.0-beta.1`.

Each package owns its own release flow (`<pkg>:patch`, `<pkg>:minor`, `<pkg>:major`), its own CHANGELOG, and its own tag pipeline that builds and publishes only its artefacts. There is no umbrella tag for the application as a whole.

Rationale for the slash separator over alternatives:

- `<pkg>/v...` (chosen): unambiguous, used by Lerna, Nx and other monorepo tooling, easy to grep/group, no collision with versions that contain dashes (alpha/beta).
- `<pkg>-v...`: rejected because `output-worker-v1.0.1` becomes ambiguous to parse and clashes visually with semver pre-release suffixes.
- `hook0-<pkg>-v...`: rejected as verbose without adding information.
- Single umbrella `vX.Y.Z`: rejected because it is the status quo we want to leave behind.

## Consequences

- Each package has its own `CHANGELOG.md` at its root, generated by `git-cliff --include-path '<package>/**'`.
- The legacy root `CHANGELOG.md` is archived as `CHANGELOG.legacy.md`. New entries land in the package-scoped files.
- The legacy SDK convention (`sdk-vX.Y.Z`) is grandfathered to avoid breaking npm/crates.io publish flows already wired to it. New SDK releases keep this prefix.
- The legacy MCP tag (`hook0-mcp-v1.0.0`) is frozen. Future MCP releases use `mcp/v...`.
- GitLab and GitHub both accept `/` inside tag names; CI rules and verification scripts parse the tag with a regex anchored on `^<pkg>/v...$`.
- Image tags published by container jobs use the version part only (e.g. `rbaumier/hook0-api:1.0.1`), not the full git tag.
